## 3. API Key 配置

默认模型兜底顺序是：

```text
DeepSeek -> Gemini
```

也就是不传 `--provider` 时，程序先调用 DeepSeek；
如果 DeepSeek 因余额、限流、网络或服务错误失败，本轮进程会自动切到 Gemini。

需要系统环境变量：

```text
DEEPSEEK_API_KEY
GEMINI_API_KEY 或 GOOGLE_API_KEY
```

如果以后切换模型，可以设置：

```text
TRADINGAGENTS_CN_LLM_PROVIDER
```

支持值：

```text

kimi
gemini
```

对应 API Key 环境变量：

```text
DEEPSEEK_API_KEY
```

对应模型名也可以用环境变量覆盖：

如果什么都不设置：

## 5. 程序会做什么

如果用户问单只股票，例如：

```text
京东方A能不能买
```

程序大致流程：

```text
用户问题
  -> 大模型路由器识别意图和股票代码
  -> LangGraph StateGraph
  -> 技术面分析员主动调用行情/技术工具
  -> 情绪面分析员主动调用社区/舆情工具
  -> 新闻公告分析员主动调用新闻/公告工具
  -> 基本面分析员主动调用公司资料/财务工具
  -> 综合汇总员汇总
  -> 多头和空头研究员结构化辩论
  -> 研究经理给出研究评级
  -> 交易员给出交易预案
  -> 风险分析员进行风险辩论
  -> 组合经理给出最终评级
  -> 最终报告
```

现在前三个 Analyst 已经不是“代码先取数据再塞 Prompt”。

现在是：

```text
分析员自己发起工具调用
LangGraph 工具节点执行工具
工具结果回到 Agent
分析员再输出报告
```

## 6. 最终报告位置

用户自然语言入口生成的报告默认保存到：

```text
outputs/user_questions
```

文件名格式：

```text
股票名_股票代码_日期_final_report.md
```

例如：

```text
京东方A_000725_2026-06-18_final_report.md
```

报告里最重要的是开头的：

```text
## 最终结论
```

它会尽量直接说明当前交易状态：

```text
可以考虑买入
小仓位试探
等待回调
加入观察池
持有观察
减仓风控
卖出回避
```

为什么不再只写“买 / 卖”？

```text
有些股票基本面不错，但短期涨太多、技术面过热。
这种情况不应该粗暴写成“卖出”，
而应该写成“等待回调”或“加入观察池”。
```

最终结论会拆成几句话：

```text
交易状态：等待回调
现在能不能买：现在不追高买入。
如果已经持有：已有仓位可先降到舒服仓位，保留观察仓。
如果还没持有：未持有者先放入观察池，等回调或企稳信号。
等待条件：等待回调到支撑位、均线、布林带中轨，或成交缩量企稳。
```

## 7. Thread ID 是什么

每次单股分析都会有一个 `thread_id`。

默认格式：

```text
single-stock-股票代码-日期
```

例如：

```text
single-stock-000725-2026-06-18
```

运行完成后终端会显示：

```text
本次 thread_id：single-stock-000725-2026-06-18
```

这个 `thread_id` 很重要。

它就是断点续跑时用来找回任务状态的编号。

## 8. SQLite Checkpoint 保存在哪里

默认 checkpoint 数据库路径：

```text
outputs/checkpoints/tradingagents_cn_checkpoints.sqlite
```

它保存的是 LangGraph 每一步的图状态。

也就是说，程序跑到：

```text
Market Agent
News Agent
Fundamentals Agent
Summary Agent
...
```

每一步都会被记录到 SQLite 数据库里。

这个数据库在硬盘上，不是内存。

所以：

```text
程序关掉后还在；
电脑重启后还在；
下次可以用同一个 thread_id 继续。
```

可以用环境变量覆盖默认数据库路径：

```text
TRADINGAGENTS_CN_CHECKPOINT_DB
```

也可以运行时指定：

```powershell
--checkpoint-db outputs/checkpoints/my.sqlite
```

## 9. 如何断点续跑

如果上一次运行中断了，例如：

```text
模型超时
网络失败
AKShare 接口失败
电脑关机
程序被手动停止
```

只要你还记得 `thread_id`，就可以继续：

```powershell
C:\software\Anaconda\envs\feishubot\python.exe .\run_user_question.py --resume --thread-id single-stock-000725-2026-06-18
```

如果使用自定义 checkpoint 数据库：

```powershell
C:\software\Anaconda\envs\feishubot\python.exe .\run_user_question.py --resume --thread-id single-stock-000725-2026-06-18 --checkpoint-db outputs/checkpoints/my.sqlite
```

程序逻辑：

```text
如果 resume=False：
    从头跑。

如果 resume=True 且 thread_id 已存在：
    从 SQLite checkpoint 继续。

如果 resume=True 但 thread_id 不存在：
    自动从头跑。
```

## 10. 什么时候需要 resume

这些情况最适合用 resume：

```text
完整单股分析跑到一半失败
深度选股分析多只股票时失败
模型 API 网络超时
模型 API 限流
AKShare 某个接口临时失败
想调试某个长链路，不想每次从头跑
```

单次快速分析如果顺利完成，不需要 resume。

### 16.6 模型没有按 JSON 格式返回怎么办

关键控制节点现在都有结构化输出重试：

```text
Router
Bull / Bear Debate
Research Manager
Trader
Risk Debate
Portfolio Manager
```

如果模型第一次没有返回合法 JSON，程序会把错误和 JSON Schema 发回模型，让它重新输出。

默认最多重试 2 次。

如果仍然失败，程序不会拿普通文本继续跑，而是返回合法兜底对象：

```text
Router:
    unknown / low confidence

Bull / Bear Debate:
    weak / 低置信度辩论观点

Research Manager:
    Hold

Trader:
    Hold

Risk Debate:
    high risk / 不允许继续交易

Portfolio Manager:
    Hold
```

这代表：

```text
模型结构化输出失败，系统进入保守兜底。
```

如果报告里出现类似文字，不代表股票一定应该持有，
而是代表这一次关键节点没有返回可校验结构化结果，系统选择不做激进交易判断。

### 16.8 多空辩论和风险辩论已经拆成真正 LangGraph 节点

当前正式 StateGraph 不再把多空辩论塞进一个 `debate_node` 里循环。

现在多空辩论节点是：

```text
summary_agent
  -> bull_researcher
  -> bear_researcher
  -> bull_researcher
  -> ...
  -> research_manager
```

路由规则：

```text
1 轮多空辩论 = Bull 发言 1 次 + Bear 发言 1 次。
max_debate_rounds = 1 时：
    bull_researcher -> bear_researcher -> research_manager

max_debate_rounds = 2 时：
    bull_researcher -> bear_researcher -> bull_researcher -> bear_researcher -> research_manager
```

风险辩论节点是：

```text
trader
  -> aggressive_risk_analyst
  -> conservative_risk_analyst
  -> neutral_risk_analyst
  -> aggressive_risk_analyst
  -> ...
  -> portfolio_manager
```

路由规则：

```text
1 轮风险辩论 = Aggressive + Conservative + Neutral 各发言 1 次。
max_risk_discuss_rounds = 1 时：
    aggressive -> conservative -> neutral -> portfolio_manager

max_risk_discuss_rounds = 2 时：
    aggressive -> conservative -> neutral -> aggressive -> conservative -> neutral -> portfolio_manager
```

为什么这一步重要：

```text
它更接近原版 TradingAgents 的 GraphSetup。
每个辩论角色都有独立节点名。
SQLite checkpoint 可以更清楚地记录程序跑到哪个辩论角色。
调试器里也能直接看到当前停在 bull_researcher、bear_researcher 或某个 risk analyst。
```

模拟盘执行链路：

```text
Portfolio Manager 最终评级
  -> process_portfolio_rating(...) 生成 TradeSignal
  -> build_risk_guardrail_decision(...) 生成风控护栏
  -> run_paper_trading_from_result(...)
  -> 读取/创建本地 PaperAccount
  -> 先复盘历史 pending 模拟成交
  -> 根据 BUY / SELL / HOLD 决定是否模拟下单
  -> 保存 account.json
  -> 写入最终报告和 full_state.json
```

BUY 的模拟执行规则：

```text
1. 必须是机器交易信号 BUY。
2. 风控护栏必须允许新开仓或允许加仓。
3. 单只股票仓位不能超过：
       min(模拟盘全局上限, 风控护栏 max_position_pct)
4. 单次买入金额不能超过风控护栏 max_single_add_pct。
5. 买入股数按 A 股 100 股整数倍向下取整。
6. 如果可买金额低于最小成交金额，记录 skipped，不下单。
```

SELL 的模拟执行规则：

```text
1. 必须是机器交易信号 SELL。
2. 只卖出模拟账户中已有持仓。
3. 当前版本不做融券卖空。
4. 如果没有持仓，记录 skipped。
```

HOLD 的模拟执行规则：

```text
不买卖。
记录一条 skipped，说明这次最终信号不是 BUY/SELL。
```
